## THE VIBE

Two sessions deep into building a real production admin dashboard for a mosque committee. The user is non-technical by framing ("non tech savy" was literally said about the GitHub commit link) but technically fluent in what they want — they catch visual misalignments, weird sort orders, and broken overflow without being told what the root cause is. They observe symptoms precisely and trust you to find the cause.

The dynamic: **they test on device, you fix the root cause, they confirm and move on fast.** No back and forth on approaches. When they say "ok good" it means done — move on. They don't want explanations unless they asked "why". When they ask "why", they genuinely want the root cause.

They build incrementally and pragmatically. When you explain a trade-off ("should we restrict months?"), they think for a second and say "ok, acceptable. dont do anything" — they don't need hand-holding to make a decision. Respect that.

**Trust level:** Very high. Fast approvals. They notice wrong values but don't micromanage approaches.

**Update from session 4:** all of the above still holds exactly. One refinement worth internalizing: the user runs on **two distinct gears**, and picks the gear implicitly — you have to read which one you're in:

- **Fast gear (default):** "add a button", "change this text", "move this note", "remove the underline" — just do it, no discussion, no options presented. Multiple rounds of these back-to-back with zero friction. Don't over-plan a one-line CSS tweak.
- **Deliberate gear (self-flagged):** when the ask touches something structural or irreversible — a new destructive bulk action (Duplicate/Clear an entire month of data), a layout restructure (ustaz two-column form) — the user *explicitly says so*: "check if my approach is good... ask me question to better understand... do not generate answer unless I allow it." When you see that phrasing, stop and actually use `AskUserQuestion` before touching any file, even if you already know what you'd recommend. They want the tradeoffs surfaced (e.g. "should Clear be extra-protected on the live month?"), pick from your options fast, then say "yes" — at which point you get a clean, uninterrupted green light to implement the whole thing in one pass.

Also notable: this user visibly *tests as they go* and comes back with precise symptom reports from real usage (not hypotheticals) — "when i click tunjukkan jadual, it shows next month" was reported from an actual click, with the exact contradictory evidence already gathered (no-param page empty, `?bulan=depan` page correct, JSON inspected). When they hand you evidence like that, trust it over your own assumptions and trace the actual data path rather than re-deriving from the code alone — reading the live `kuliah/data/jadual_lengkap_beta.json` mid-conversation is what turned "why is this happening" into a five-minute diagnosis instead of a guessing exercise. Session ended on a satisfied, low-key "ok good, now it fixed" — that's the full signal that a fix landed; no further confirmation needed once you hear it.

- **Vercel** — hosts the static site + `api/publish.js` serverless function. Sole deployment target as of session 7 — GitHub Pages was retired, don't reintroduce Pages-specific assumptions anywhere.
- **No frameworks** — pure HTML5/CSS3/Vanilla JS, zero npm

### Session 3 (2026-07-06 — print/PDF export bug fix, from the repo-root chat, not admin)

This shell is **plain static HTML+CSS, identical on every page, zero JS/CDN dependency** — a first pass built it via JS + Tailwind Play CDN and that caused real flicker on this multi-page app (the CDN re-scans/regenerates CSS on every single full-page navigation, not just once), so it was reverted to hand-written CSS the same session it shipped. **Don't reach for Tailwind Play CDN anywhere else in this repo's MPA pages — the "no build step" pitch is a per-navigation tax here, not a one-time cost.** Only `#sidebar-nav`'s *contents* (the actual links) are JS-rendered, by `renderSidebar()` in `app.js` from a single `MODULES` config array (module key, `permission` string OR `null` for "visible to any authenticated admin, no gate", `requiresSuperAdmin` flag, items with `href`+`match` pathname arrays). **Add a module or page by adding one `MODULES` entry — that's the whole change now**, not touching multiple functions across multiple pages like the old flat-nav era required. `admin/dashboard.html`'s View Transitions API (`@view-transition { navigation: auto; }`) smooths cross-page nav further, purely additive, degrades silently on unsupported browsers.

**Mobile breakpoints:**
- `≤768px` — tablet: compact padding, smaller elements; also where the sidebar goes off-canvas (`.sidebar` gets `transform: translateX(-100%)` by default, `.sidebar.open` reverses it, toggled by `toggleNav()`/`closeNav()`)
- `≤640px` — phone: hamburger, card tables, day list (`admin/`); `kuliah/jadual/` (and the archived `kuliah3/jadual/`) uses `≤768px` as its own mobile/desktop split (different file, different breakpoint — don't confuse the two)

**Any `@media (max-width: ...)` block must be `@media screen and (max-width: ...)`:**
Without `screen`, mobile layout rules stay active during print/PDF export on narrow devices and `@media print` can't reliably reset them. Broke PDF export this way (session 3, in what was then `kuliah3/jadual/` — that file is today's `kuliah/jadual/style.css` after the session-7 folder swap) — applies to any new print/export feature in either `kuliah/jadual/` or the archived `kuliah3/jadual/`.

**Card-per-row table pattern:**
Every `<td>` in JS-rendered tables must have `data-label="..."`. Empty label `data-label=""` is fine for action columns (no prefix shown).

**Ustaz sort is client-side, everywhere it's used:**
Never add `.order()` back to the Supabase ustaz query — it breaks numeric sort. Sort with `localeCompare({numeric:true, sensitivity:'base'})` in JS after fetch. **Email matching against `admins.email` must use `.ilike()`, not `.eq()` (session 7 discovery):** Postgres `text` equality is case-sensitive by default, and there's no guarantee `admins.email` (often typed by hand, e.g. via the `setup.sql` §5b bootstrap insert) matches the exact casing Google OAuth/Supabase Auth actually returns for that account. A mismatch doesn't error — `.eq('email', ...)` just silently returns zero rows, and whatever fallback exists kicks in quietly (in this case, falling back to displaying the raw email instead of the registered name). Both `jadual.js`'s (renamed from `dashboard.js` session 12) `loadLastPublishedNote()` admin-name fallback lookup and `api/publish.js`'s own actor-name lookup were switched to `.ilike('email', ...)` / `email=ilike.` for this reason. **Use `ilike` for any future email-matching query against `admins` in this repo** — `admin/dashboard.js` (the session-12 cross-module overview, a DIFFERENT file from `jadual.js` despite the similar name) follows the same rule in its own `loadKuliahLastPublished()`.

**Write access is enforced in Postgres now, not just in the UI (session 13, `setup.sql` §9) — but re-running §9 wholesale will error, know this before you paste it a second time.** `admin_can_write(module_key)` and `admin_is_super_admin()` (`SECURITY DEFINER` SQL functions, `SET search_path = public`, matched to the current admin via `email ILIKE auth.email()` for the same casing reason as the `.ilike()` rule below) back real per-table SELECT-open/INSERT-UPDATE-DELETE-gated RLS policies on `admins`/`ustaz`/`schedule`/all 4 infaq tables — client-side `canWriteModule(moduleKey)` (`app.js`) mirrors this exactly but is UI-only defense-in-depth, not the real boundary. **`§9`'s `CREATE POLICY` statements for the NEW policy names have no matching `DROP POLICY IF EXISTS` guard** (only the OLD single-permissive-policy names get dropped) — re-pasting the whole section a second time will fail on the first `CREATE POLICY "auth_select_admins"` (etc.) with "already exists," and since a multi-statement pasted query runs as one implicit Postgres transaction, that failure rolls back the *entire* paste, including anything unrelated bundled into the same run. If a future migration needs to touch these policies again, either target the specific new lines that changed, or add proper `DROP POLICY IF EXISTS` guards for the new names too before re-running the whole block. **`activity_log`/`infaq_activity_log` were deliberately left OUT of this — still the old single-permissive-policy shape, on purpose** (logging is fire-and-forget from every role; gating it risked breaking a legitimate editor's own audit entry for the sake of blocking a `viewer`'s harmless fake log row). Don't assume every table got the same treatment — check `setup.sql` §9's actual table list before assuming a given table is write-gated.

**Any new Supabase table needs explicit `GRANT`s, not just an RLS policy (session 6 discovery — hit twice, in production, same session):** `CREATE POLICY ... TO authenticated` is necessary but NOT sufficient — Postgres checks table-level `GRANT` privileges before RLS is even evaluated, and unlike `ustaz`/`schedule` (which happened to inherit grants from this Supabase project's default privileges when they were originally created), a brand-new table does not automatically get them. `activity_log` hit `permission denied for table activity_log` from the browser (missing `GRANT ... TO authenticated`), then hit the *identical* error again for `api/publish.js`'s server-side insert specifically (missing `GRANT ... TO service_role` — a different Postgres role than the browser uses). **Any future new table needs BOTH grants added explicitly in `setup.sql`** — don't assume either is inherited, and if a fresh table throws "permission denied," check this before anything else.

**Activity log (`activity_log` table + `logActivity()` in app.js, viewed at `userlog.html`, super_admin only — built session 6):** every mutating admin action (schedule day-edits, bulk duplicate/clear as one summary row each, ustaz CRUD, admin-account CRUD, Terbitkan/publish) inserts one row right after its write succeeds. `target_label`/`detail` are always plain-text snapshots — never a live FK — so history survives later renames/deletes of the thing being referenced. Each call-site captures "before" from an already-loaded cache (`scheduleMap`/`allUstaz`/`allUsers`), never an extra query, and skips the insert on true no-op saves. `logActivity()` never throws/toasts — a logging failure must never look like the actual save failed. See Session 6 above for the full build story, the GRANT landmine (previous bullet), and the run of "should super_admin be able to delete/annotate entries" design questions that were deliberately answered but NOT implemented — don't build a delete or note feature for this table unless the user explicitly asks again; last read on both was "not worth it yet."

**Never `await` a third-party API inside a render path without a timeout (session 9 discovery — this took down a whole page section in production):** the mobile day-card list on `kuliah/jadual/` vanished because `initializeMobileView` → `renderTodayCard` → `loadHijriDate` → `fetch(e-solat)` was one serial await chain, and the e-solat government API hung for 30+ seconds with no timeout — so a purely cosmetic Hijri-date lookup blocked the entire month list. The standing pattern now in `kuliah/jadual/script.js`, reuse it for any future external call: **(a)** `fetchWithTimeout()` — `AbortController` + 5s deadline, **(b)** fire-and-forget from the render path (never `await` cosmetic data before rendering primary content), **(c)** a local fallback (`gregorianToHijri()` calculator), **(d)** a request-id guard (`hijriRequestId`) so a slow stale response can't overwrite a newer selection, and **(e)** re-query the target element at write time — an element reference captured before an `await` may be a detached node if the UI re-rendered meanwhile. Related: **prefer `api.waktusolat.app` over `www.e-solat.gov.my` for anything JAKIM-sourced** — same data, community mirror, ~0.4s and CORS-friendly vs a direct endpoint that demonstrably hangs; `waktu-solat/` has always used it, `kuliah/jadual/` switched in session 9, `calendar/hijri/` still hasn't (flagged, not done).
